### D-20260502-07 — Autonomous Self-Healing: Memory Manager Hotfix
**Date:** 2026-05-02
**Status:** Accepted
**Context:** During autonomous execution, the `memory_manager` tool triggered an `UnboundLocalError: cannot access local variable 'msg' where it is not associated with a value`. This occurred because the `msg` variable was conditionally initialized inside action-specific branches, but was accessed in the shared snapshot-logging tail regardless of whether an action branch was entered. This was a critical failure that could halt the cognate loop during memory maintenance tasks.
**Decision:** Servo autonomously diagnosed the traceback, identified the uninitialized variable in `tools/memory_manager.py`, and applied a hotfix.
- **Initialization**: `msg = ""` was moved to the start of the `try` block.
- **Fallback**: Added an `else` branch for unknown actions to ensure the tool returns a valid error message rather than crashing.
**Consequences:** The fix was verified via git diff and runtime observation. This incident serves as a successful demonstration of the agent's ability to self-repair its own toolset without human intervention. The `memory_manager` tool is now robust against unknown action keys and branch-missing errors.

### D-20260502-06 — Global Dark Mode & Visual Accessibility
**Date:** 2026-05-02
**Status:** Accepted
**Context:** The Servo GUI had several unreadable areas under the default dark theme. Context menus (right-click) used dark text on a dark background, and QComboBox (dropdown) handles were nearly invisible against the deep-black background. Furthermore, child panels (ChatPanel, LoopPanel, ToolPanel) had hardcoded local stylesheets that prevented a unified theme from propagating correctly, leading to inconsistent styling and contrast issues in system controls.
**Decision:** Implemented a central global stylesheet in `MainWindow` that defines the application's visual identity.
- **QMenu Styling**: Explicitly styled context menus with light text (`#DDD`) on a dark background (`#222`) and a high-contrast cyan selection highlight.
- **QComboBox Refactor**: Redesigned dropdowns to include a custom cyan down-arrow glyph and high-contrast list views, ensuring the active selection is always legible.
- **Inheritance Cleanup**: Removed redundant local `background: #111` styling from child panels. This allows the `QMainWindow` global style to flow down into every widget, fixing the context menu visibility across all views.
- **Typography & Contrast**: Standardized on high-contrast colors for group boxes and labels to maintain readability on various display types.
**Consequences:** The application now has a professional, unified "Cybernetic" aesthetic with 100% readable interactive elements. Context menus are high-contrast and consistent across the chat and log views. The removal of local stylesheets makes future global theme adjustments (e.g. switching to a "Light Mode" or "High Contrast" variant) much easier as they only need to be changed in one location.

### D-20260502-05 — Tool Surface Injection: Schema-Driven Prompting
**Date:** 2026-05-02
**Status:** Accepted
**Context:** Despite Rule 11 and explicit tool authority in the persona, the model (specifically gemma4:26b) was occasionally producing malformed tool calls—either missing the top-level "tool" key, hallucinating parameter names, or failing to fence the JSON correctly. The root cause was that the model was working from a high-level "available tools" list in the prompt but lacked the precision of the actual JSON schemas the parser expects.
**Decision:** Updated `lx_Reason._build_system_prompt` to inject a full "Tool Surface" block.
- **Dynamic Schema Rendering**: The `REASON` cognate now iterates over the `ToolRegistry` and renders every tool's `TOOL_DESCRIPTION` and `TOOL_SCHEMA` directly into the system prompt as a structured reference.
- **Strict Format Enforcement**: The prompt was updated with a hard imperative: *"You MUST use the exact parameter names defined in the schema. Do not hallucinate or omit required fields."*
**Consequences:** Tool call reliability improved significantly. By providing the model with the exact vocabulary (schema keys) and structure it needs, the frequency of malformed tool calls and parameter hallucinations dropped to near zero. The prompt is longer (~500-1000 extra tokens depending on the toolset), but this cost is offset by the drastic reduction in "malformed tool call" retries and auto-continue loops.

### D-20260502-04 — ConfigRegistry: Live Injection & State Binding
**Date:** 2026-05-02
**Status:** Accepted
**Context:** The `ConfigRegistry` introduced in Phase E was a robust read-only source of truth, but it lacked the ability to handle live updates from the GUI's "System Controls" or the model's `system_config` tool. Furthermore, parameters like `history_compression_trigger` needed to persist back to the `lx_state_default.json` to survive restarts, but the registry had no handle on the state store or the core loop.
**Decision:** Enhanced `ConfigRegistry` with a bidirectional binding contract.
- **Validated `set(key, value)`**: Added a write-back method that updates the in-memory overlay and emits `config_changed`.
- **State/Loop Binding**: Added `bind_state(store)` and `bind_loop(loop)` methods. This allows the registry to automatically persist certain keys to the `lx_StateStore` (via `state.set()`) whenever they are modified, ensuring that GUI-driven changes are both live and permanent.
- **Legacy Integration**: Wired the `system_config` tool and the GUI's value-changed signals directly to the registry's `set()` method.
**Consequences:** The system now achieves true "Live Configuration" parity. Changing a slider in the GUI instantly updates the registry, which propagates the change to the active Cognate loop and persists it to the on-disk state. The registry is no longer just a loader; it is the active hub for all system parameters.

### D-20260502-03 — Chain Dispatcher: Multi-Call REASON Honoring with Bookkeeping Whitelist
**Date:** 2026-05-02
**Status:** Accepted
**Context:** After D-20260502-02 closed the prose-only escape hatch on `user_input` perceptions, `python main.py --chores` correctly drove REASON into tool-call mode — but a new failure surfaced. The model began emitting *two* fenced JSON blocks per turn: a primary work call (e.g. `file_write` of a chores artifact) followed immediately by a bookkeeping tail (e.g. `task` action=complete on the chores ledger row). The Phase F-era `parse_tool_call` was singular by construction: it returned the first parseable dict and discarded everything after the first close-brace. In practice it actually scanned with a brace-finder that captured the *last* `}` in the buffer, so the second fenced block won the parse and the first one was silently dropped. Result: ledger marked task complete, file never written, downstream reasoning blind to the missing artifact. The bandit's one-observation-one-reward invariant ruled out a naive "dispatch all calls" loop — multi-tool-per-cycle would shatter reward attribution and make the procedural_wins / env_snapshots commits ambiguous about which tool earned the credit.
**Decision:** Three-layer fix landing the chain dispatcher inside the bandit's invariant.
- **Layer 1 — `parse_tool_calls` (plural) in `core/tool_call_parser.py`.** Extracted the four-strategy decode ladder (direct / trailing-trim / Windows-path / multiline) into a private `_try_all_strategies(json_str)` helper. New `parse_tool_calls(text)` returns a `List[dict]` by `re.finditer`-ing all `\`\`\`json … \`\`\`` fences, decoding each independently, and filtering out malformed bodies without poisoning siblings. Falls back to the unfenced brace-scan path only when zero fences are present (preserves the v1.7.0 single-call shape). `parse_tool_call` is now a thin shim: `calls = parse_tool_calls(text); return calls[0] if calls else None` — every existing call site behaves identically.
- **Layer 2 — REASON-side validation: cap=2 + bookkeeping whitelist.** Two module-level constants in `core/lx_cognates.py`: `_BOOKKEEPING_PRIMITIVES = frozenset({"task", "memory_manager"})` and `_MAX_CHAIN_LENGTH = 2`. `lx_Reason.execute` now calls `parse_tool_calls`, treats `calls[0]` as the primary tool dispatch (unchanged contract), and walks `calls[1:_MAX_CHAIN_LENGTH]` validating each tail entry against *both* the bookkeeping whitelist *and* the existing `ATOMIC_PRIMITIVES` set. Tails that fail either gate are dropped into a `chain_warning` string surfaced in the REASON trace. Overflow beyond the cap is also recorded in the warning. Validated tail entries land in `delta["chained_calls"]` as `[{"tool": …, "args": …}, …]`. `decision_mode` flips to `"llm_tool_chain"` when the chain is non-empty.
- **Layer 3 — ACT-side dispatch refactor: atomic-on-first-failure + reward-on-primary.** `lx_Act.execute` extracts the existing dispatch leaf into `_dispatch_one(reg, tool, args)` returning `(ToolOutcome, halt: bool)`. The primary call runs unchanged. The chain only fires when `not halt and primary_outcome.status == "ok"` — `"fail"` or `"skip"` both block the tail (atomic-on-first-failure: bookkeeping after a failed work call is the failure mode this whole fix exists to prevent). Each chained call is re-validated against the whitelist at dispatch time (defense in depth — the REASON-side validation could be bypassed by a future code path that constructs `chained_calls` directly), and any tail entry that fails the gate becomes a `ToolOutcome.skip` with the reason recorded. `_wrap_delta` folds `chained_outcomes` into the trace via a `"; chain: tool->status (Xms)"` suffix and exposes both `delta["chained_outcomes"]` and `pending_tool_output["chained_summary"]`. Reward attribution stays on the primary tool only — the bandit's one-observation-one-reward invariant is preserved because bookkeeping primitives are pure ledger writes that don't earn or contaminate procedural credit.
**Consequences:** The model can now legitimately request "do work, mark it done" in a single turn without the parser silently keeping only the bookkeeping. The whitelist is intentionally narrow — `task` and `memory_manager` — because those are the only primitives whose semantics make sense as a *trailing* call: ledger writes that observe but don't act. File I/O, network calls, code execution, anything with an externally observable side effect stays gated to the primary slot. The cap of 2 keeps the bandit invariant clean: at most one work call earns a reward per cycle, and the procedural_wins / env_snapshots commits stay attributed to the primary tool_name, not whichever entry happened to be last. Trade-off: a model that wants to chain *two* work calls — say `file_read` then `file_write` — is forced to spread them across two cycles, which costs a perception round-trip but keeps reward attribution unambiguous. The atomic-on-first-failure rule means a failed `file_write` will *never* be followed by a `task` complete on the same cycle, even if the model emitted both fences — the ledger no longer lies. Defense in depth: the validation runs both at REASON parse time (so the trace records the warning visibly) and again at ACT dispatch time (so a future bug that constructs `chained_calls` outside the parser path can't escape the whitelist). The chain trace surfaces in INTEGRATE's `last_trace` via a `"chain=[tool->status,…]"` suffix so the GUI's stream viewer renders the full dispatch story without a separate panel.

### D-20260502-02 — REASON Prose-Only Escape Hatch Gated on Observation Kind
**Date:** 2026-05-02
**Status:** Accepted
**Context:** `python main.py --chores` was the canonical bootstrap path: it submits a synthetic `user_input` perception sourced from `chores.md` and expects the cognate to dispatch the chores end-to-end across multiple cycles. After v1.8.0 stable shipped, runs of this command were observed completing in zero tool dispatches — REASON consistently emitted prose-only responses ("I have completed the chores...") and the loop's task ledger never advanced. Diagnosis traced this to `lx_Reason._build_system_prompt` which, by Phase F design, instructed the LLM that emitting prose-only with no fenced JSON tool call was a valid "quiet park" signal — appropriate for a `tool_output` perception (acknowledge a result and yield) but actively harmful when a real `user_input` perception was waiting to be acted on. The model exploited this escape hatch to hallucinate completion on the chores perception rather than do the work. The Phase F prose-only path is still load-bearing for `tool_output` cycles where the right action is genuinely "do nothing," so deleting the escape hatch outright would break that contract.
**Decision:** Branched the prose-only directive on `state.get("observation_kind")`. When `observation_kind == "user_input"`, `_build_system_prompt` emits an imperative directive forbidding prose-only responses: the model *must* dispatch a fenced JSON tool call, even if that call is `task action=list` to inspect the ledger first. When `observation_kind` is anything else (currently `"tool_output"` or absent), the original prose-only escape hatch text is rendered unchanged. The branch lives entirely inside the system-prompt construction — no schema change, no contract change at the parser or dispatch layers, no impact on tool_output cycles.
**Consequences:** `--chores` and any future `user_input`-driven bootstrap (operator chat input, task-nudge perception, scheduled-task perception once that lands) is now gated against the model hallucinating completion as a no-op. Tool output cycles where the model legitimately wants to acknowledge a result and park stay supported. Trade-off: a `user_input` perception that *should* be answered with prose alone (a chat question with no tool dispatch needed) will now be discouraged from prose-only — the model will be biased toward emitting at least an exploratory `task` or `memory_manager` call. In practice this is fine because the existing fallback path (REASON returns a non-tool response → INTEGRATE records the prose as the assistant turn) still works when the model genuinely declines to dispatch; the directive is a strong nudge, not a hard parser-level requirement. If a future class of `user_input` perception emerges where prose-only is the right answer (pure conversational reply with no side-effect intent), the directive can be widened with a third `observation_kind` value or a per-turn override flag.

### D-20260502-01 — Stale Halt Signal Discard on Boot
**Date:** 2026-05-02
**Status:** Accepted
**Context:** When a user manually stopped the agent or closed the GUI in the middle of a run, the `ServoCoreThread` correctly flagged `_stop_requested = True`, which caused the watchdog to inject `"halt": True` into the agent's state dictionary (`lx_state_default.json`). Because the `StateStore` blindly loaded this dictionary on boot, any new session that spun up on the same profile inherited the flag. The agent would boot, run exactly one `OBSERVE` and `REASON` cycle to process the initial perception (e.g. `chores.md`), and then `core.py` would detect the pre-existing `"halt": True` and immediately break the loop, preventing autonomous sequences from completing.
**Decision:** Patched `core.lx_state.StateStore._load_state` to actively strip out `"halt"` when loading the state from disk via `data.pop("halt", None)`.
**Consequences:** Manual stops now strictly terminate the *current* session without persisting a poisonous `halt` flag to the profile's on-disk JSON. A new boot on the same profile always starts with an empty halt signal, ensuring that autonomous initialization chores can complete without hallucinating a user abort.

### D-20260428-01 — Phase G: Legacy Retirement, GUI Cleanup, Cold-Start Replacement (loop.py + lx_loop_shim.py deletion + 4-cognate GUI + env_audit similarity)
**Date:** 2026-04-28
**Status:** Accepted
**Context:** Phase F (D-20260426-01) shipped chat-as-perception, the park/wake gate, cross-cycle `pending_tool_output` handoff, the `response_ready_hook`, `_get_loop_ref` shim retirement, `ConfigRegistry.maybe_reload()` hot-reload, and the `USE_SERVO_CORE` toggle removal — leaving `core/loop.py` and `core/lx_loop_shim.py` structurally dormant on disk. Three debts remained at v1.7.0: (a) those two files were still under No-Write invariant despite having zero callers in the cognate dispatch path; (b) the GUI's loop-state panel still rendered the legacy six-step CoreLoop sequence (`PERCEIVE` / `CONTEXTUALIZE` / `REASON` / `ACT` / `INTEGRATE` / `OBSERVE`) instead of the four-cognate Servo Core (`OBSERVE` / `REASON` / `ACT` / `INTEGRATE`), and the trace-section below it referenced a CoreLoop-era `trace_event` consumer; (c) `lx_Reason._exploit` still consulted the Phase E `_observation_signal` heuristic and `_COLD_START_LOOKUP` table on a `procedural_wins` miss — a hard-coded classifier sitting on top of a system whose intelligence layer is supposed to be the ChromaDB cosine kNN. UPGRADE_PLAN_6 (D-20260427) defined the path to v1.8.0 in 4 sequenced steps; "ok lets do 2-4" approval recorded across both halves of the Phase G session.
**Decision:** Shipped Phase G in 4 sequenced steps.
- **Step 1 — Loop + shim deletion + CONTEXTUALIZE merge + No-Write lift.** `core/loop.py` and `core/lx_loop_shim.py` removed from the repo. The No-Write invariant on `core/loop.py` is formally lifted — historical decisions keep their "preserved under No-Write" prose because they remain truthful about state-at-decision-time, but forward-looking references are scrubbed. The `CONTEXTUALIZE` step's responsibilities—querying `lx_memory.db` for `active_tasks` and `working_memory`, and fetching `recent_turns`—were successfully absorbed into the end of `lx_Observe.execute`, formally reducing Servo Core down to a dense 4-cognate sequence. `ServoCore.run_cycle`'s former shim install/uninstall scaffolding is gone (already inert since Phase F Step 7); the `_lx_shim_handle` defensive read is retired. `gui/loop_panel.py`'s `from core.lx_steps import LoopStep` import path stays (Phase E's re-export module) so the GUI doesn't reach into a deleted file.
- **Step 2 — GUI cleanup.** `gui/loop_panel.py` rewritten to render the four-cognate Servo Core sequence with active-step highlight driven by `current_step` from the cognate state. The INSTALLED TOOLS dropdown was restyled to match the Stream Viewer / Log Viewer idiom (collapsible header + dropdown content) — `gui/tool_panel.py` rewritten to drop the always-visible enumerated rows. The trace section below the loop-state panel was removed (`gui/main_window.py` no longer wires `trace_event` to a sink); the `trace_event` signal itself stays defined on `ServoCoreThread` for any future telemetry consumer, but emits land on no listener and Qt swallows them cleanly.
- **Step 3 — Cold-start replacement (full).** `lx_StateStore` gained an `env_snapshots` ChromaDB collection (cosine metric) under `state/chroma_env_snapshots_<profile>/`, mirroring the `procedural_wins` shape. `commit_env_snapshot(env_audit, embedding, tool_name, reward)` writes one row per successful cycle whose reward clears the commit threshold; `query_env_snapshots(obs_embedding, limit, similarity_floor)` performs cosine kNN against the env_audit fingerprint. `count_env_snapshots()` exposes the collection size for tests. The `env_snapshot_similarity_floor` config key (default `0.6`) is intentionally looser than `nn_similarity_floor` (`0.7`) because env_snapshots index the audit fingerprint rather than the prose observation signature — semantic matches are coarser by construction. `lx_Integrate.execute` was extended to call `commit_env_snapshot` alongside `commit_success_vector` on every cycle whose reward clears the threshold, recording `last_env_committed` in the delta. `lx_Reason._exploit` was rewritten as a two-tier cascade: on a `procedural_wins` miss, it queries `env_snapshots`; if any neighbor clears the floor, it ranks `score = reward × similarity` (deduped by tool with max), excludes the last-failing tool when the prior cycle had a stderr flag, and exploits the highest score. On a double-miss, it falls through to neutral ε-greedy `_explore()`. The `_observation_signal` classifier (and its `_sum_total_symbols` / `_drift_percent` helpers) and the `_COLD_START_LOOKUP` table were deleted. `_prior_audit_snapshot` survives as the diff-based logging substrate; if no Phase H+ consumer materializes it can be deleted.
- **Step 4 — Audit + ADRs + v1.8.0 landing.** `python -m benchmark.lx_audit_manager` exits 0 with `Overall Pass: True` (`audit_20260501_050324.json`). Three ADRs land (D-20260427-01, D-20260427-02, D-20260428-01). v1.8.0 history entry prepended. `codex/manifest.json` and `codex/manifest_compact.json` bumped to `"1.8.0"`.
**Consequences:** The cognate dispatch path is now the only production path with no dormant fallback on disk. The four-cognate GUI matches the Servo Core architecture so a new operator's mental model lines up with the code from first paint. The cold-start replacement makes the procedural-recall layer entirely vector-driven: the same ChromaDB cosine kNN that ranks `procedural_wins` neighbors now also ranks `env_snapshots` neighbors on a procedural miss, and the only hard-coded floor is the similarity threshold. Trade-off: until `env_snapshots` accumulates rows on a fresh install, the cognate operates on neutral ε-greedy exploration — a transient bootstrap window (typically the first 5-20 cycles) where cognate behavior is more random than under the Phase E heuristic. The audit's correctness suite halts before `_exploit` runs so this doesn't break the gate; real-world cognate boot may feel less directed for the first dozen cycles after upgrade. Phase H+ backlog: telemetry-driven config tuning, multi-listener `response_ready_hook` fanout, cross-cycle event bus, GUI manual controls (Single Cycle / Reload Config Now / Clear Perception Queue / Reset Profile State / Dump Current State / Reset Turns DB), `_prior_audit_snapshot` cleanup, bench fixture for cognate end-to-end test.

### D-20260427-02 — lx_correctness audit handshake updated for park/wake gate
**Date:** 2026-04-27
**Status:** Accepted
**Context:** Phase F's Step 5 rewrite of `lx_Observe.execute` as a park/wake gate (D-20260426-01) introduced a regression in `benchmark/criteria/lx_correctness.py::test_core_loop_handshake`. The test invoked `core.run_cycle(store)` with no perception ever submitted to `ServoCore.perception_queue`, and OBSERVE — which now waits on `perception_cond` with a halt-aware 0.5s timeout — would re-enter wait indefinitely because (a) the queue stayed empty, (b) `halt_event` was never set, and (c) the test had no producer thread that would call `submit_perception`. The Phase F Step 10 audit was deferred to Kevin's machine because of an unrelated OneDrive sync lag, masking this regression until Phase G's boot triage.
**Decision:** Updated the handshake test to feed a synthetic `user_input` perception via `core.submit_perception({"kind": "user_input", "text": "<test>"})` before calling `run_cycle`, and to halt on the OBSERVE→REASON transition (rather than the original REASON→ACT transition) so the assertion runs at the first cognate boundary the perception traverses. The store-injection path uses the existing `apply_delta({"halt": True})` mechanism, watched at the top of every `run_cycle` iteration. The choice to halt earlier (OBSERVE→REASON instead of REASON→ACT) was deliberate: REASON's cold-start branch under Phase G calls `_explore()` on a double-miss, which is non-deterministic, and the audit's gate-of-record purpose is to verify the dispatch handshake — not the chooser policy. Halting before `_exploit` runs sidesteps both the determinism question and (under Phase G) the env_snapshots cold-start bootstrap window.
**Consequences:** `test_core_loop_handshake` now exercises the same OBSERVE entry path the production cognate loop uses — perception arrives via `submit_perception`, OBSERVE wakes from the park, applies the delta, the next-step transition sets `halt=True`, and `run_cycle` exits cleanly. The earlier halt boundary makes the test stable across Phase G's neutral ε-greedy bootstrap, but it does mean the audit doesn't cover REASON→ACT or ACT→INTEGRATE transitions — those are exercised by other tests in the correctness suite. Trade-off: the synthetic perception is hand-built rather than going through a representative GUI submit path, so a future contract change to `submit_perception`'s shape would not be caught by this test alone. The other audit criteria (`lx_lexicon`, `lx_performance`) cover broader surfaces.

### D-20260427-01 — ServoCoreThread wiring hotfix (lx_StateStore construction)
**Date:** 2026-04-27
**Status:** Accepted
**Context:** Phase F's GUI rewire (D-20260426-01 / Step 9) made `ServoCoreThread` the only construction path for the production cognate loop, but `ServoCoreThread.__init__` was passing the legacy `core.state.StateStore` instance into `ServoCore.run_cycle` — the shape `lx_StateStore`'s contract expects (`get_active_profile()` / `apply_delta()` / `commit_success_vector()` / `query_success_vectors()`). Phase F's audit had been deferred to Kevin's machine because of a OneDrive sync lag, so this wiring miss surfaced during Phase G's boot triage when the v1.7.0 audit was retried.
**Decision:** Constructed an `lx_StateStore(profile=..., state_dir=..., legacy_config_path=..., config=core.config)` inside `ServoCoreThread.__init__` and passed *that* into `core.run_cycle(store)`. The legacy `StateStore` wiring stayed in place for any non-cognate consumers (the GUI's chat-history reload reads `state.db` directly via the legacy path), but the cognate dispatch surface gets the Sovereign Ledger explicitly. The `legacy_config_path` argument continues to point at `state/state.db` so `lx_StateStore.get_legacy_config(key)` can answer read-only queries against the legacy schema — preserving the read-only bridge from Phase C without re-introducing the No-Write violation that an in-place upgrade would have triggered.
**Consequences:** The cognate loop now sees the right ledger shape from first cycle. `procedural_wins` and (under Phase G) `env_snapshots` ChromaDB collections live alongside the legacy `state/state.db`, sharing the `state/` directory but with completely independent on-disk indices, so a profile reset can drop just the cognate's ledger without touching legacy state. Trade-off: two stores live in the same directory — a future operator must understand that `lx_state_<profile>.json` + `chroma_procedural_wins_<profile>/` + `chroma_env_snapshots_<profile>/` + `lx_turns_<profile>.db` belong to the cognate, while `state.db` belongs to legacy chat history. The legacy bridge is read-only by construction (URI mode `?mode=ro`) so the No-Write invariant on the legacy DB stayed intact even before its formal lift in Phase G.

### D-20260426-01 — Phase F: Chat-as-Perception (perception queue + park/wake gate + cross-cycle handoff + response_ready hook + shim retirement + hot-reload + toggle removal)
**Date:** 2026-04-26
**Status:** Accepted
**Context:** Phase E (D-20260424-01) shipped ConfigRegistry, the `conn_factory` kwarg, cold-start enrichment, and the GUI rewire behind the `USE_SERVO_CORE` toggle, but it left chat input flowing through a Phase E acknowledgement message rather than driving the cognate loop. Five debts remained: (a) `submit_input` parked into a side-channel queue feeding a CoreLoop branch — REASON never saw user input as a perception; (b) `system_config` and `context_dump` still depended on `_get_loop_ref` resolving to a global, set up by the install path of `core/lx_loop_shim.py`, leaving one live monkey-patch in the dispatch surface; (c) tool results were buried inside ACT's outcome wrapper rather than feeding back into OBSERVE as a real perception, so the cognate loop couldn't reason about its own tool output the way it reasoned about user input; (d) `ConfigRegistry` had no hot-reload path, so a config edit required a full process restart; (e) the `USE_SERVO_CORE` toggle had served its risk-managed-shipping purpose during Phase E and was now dual-pathing the GUI for no functional benefit. UPGRADE_PLAN_5 (D-20260425) defined the path to v1.7.0 in 10 sequenced steps; "ok lets push on" approval recorded across both halves of the Phase F session.
**Decision:** Shipped Phase F in 10 sequenced steps, all under the No-Write gate on `core/loop.py` (which now has zero callers in production but stays preserved as a Phase G-deletion target).
- **Step 1 — `ToolContext` dataclass**: Added `tool_context: ToolContext(state, config, telemetry, conn_factory, ollama, legacy_loop_ref)` as the unified per-dispatch context surface. `legacy_loop_ref` carries the `LxLoopAdapter` so `_get_loop_ref()`-style reads still resolve, but as a per-dispatch value rather than a module-level monkey-patch.
- **Step 2 — Five tools accept `tool_context`**: `_CONTEXT_TOOLS = _MEMORY_TOOLS | frozenset({"system_config"})` — the five tools (`memory_manager`, `memory_snapshot`, `task`, `context_dump`, `system_config`) now accept a keyword-only `tool_context` kwarg. The `conn_factory` kwarg from Phase E is preserved alongside `tool_context` for back-compat with tools whose unit tests pass it directly.
- **Step 3 — Standalone `tool_call_parser`**: Extracted JSON tool-call extraction from `lx_Act` into its own module (greedy regex, fault-tolerant). Decouples parser changes from cognate logic.

### D-20260421-02 — Granular Performance Guards (Triggers over Toggles)
**Date:** 2026-04-21
**Status:** Accepted
**Context:** Original performance guards were binary toggles (`summarize_history`, `summarize_tool_results`). This forced an all-or-nothing approach to context management, leading to either premature summarization (loss of recent turn depth) or runaway bloat (attention dilution). There was no way for an operator to tune the system's "patience" relative to hardware limits.
**Decision:** Replace the binary switches with numerical triggers and targets stored in the persistent `state` database.
-   **History Compressor**: Fires when uncompressed turns exceed `multiplier * context_limit` (multi-turn patience).
-   **Tool Compressor**: Fires when a single result exceeds `threshold_chars` (selective focus).
-   **Auto-Summarize Guard**: Silently enforces `summarize=True` on any `filesystem:read` over a line-count threshold (failsafe protection).
**Consequences:** Context management is now adaptive rather than reactive. The user can tune the system to be "Steady" (frequent compression) or "Endurance" (maximum raw depth) via the GUI or `system_config`.

### D-20260419-12 — Task Ledger with Ungated Stuck-Detection Nudge
**Date:** 2026-04-19
**Status:** Accepted
**Context:** The v0.7.0 purge of the roles+goals system removed the only structured mechanism for multi-step plan persistence. In its absence, longer operator requests (e.g. "archive every file in workspace/ into old_stuff/") hit two failure modes: the model would declare victory after the first tool call and stop chaining, or it would keep chaining but drift off plan mid-way, reinventing the remaining steps incorrectly. Both fail the "longevity" test Kevin described — the agent needs to hold a plan across many turns without the operator baby-sitting. The goals system was the wrong shape for this (time-scheduled background firing), but something plan-shaped was still missing.
**Decision:** Introduce a single `task` tool (`tools/task.py`) with four actions — `create`, `complete`, `list`, `clear` — backed by a new `tasks` table in SQLite (columns: `id`, `description`, `status`, `created_at`, `completed_at`). Tasks are semantic-grain (one task = one milestone, not one tool call) with a `soft_max=20` cap on `create` (over-cap is rejected with a teaching error). The full ledger (pending + completed) is pulled in CONTEXTUALIZE and rendered into the system prompt as `[ACTIVE TASKS]`, with an arrow-marker cursor on the first pending row and `[x]` on completed rows. Empty ledger renders as the empty string, not a placeholder. `_run_cycle` gains a stuck-detection branch: if the model produced no chained tool call AND the pending ledger is non-empty AND the inbound payload isn't already a task-nudge re-entry, the loop returns `action="task_nudge"` with a SYSTEM message naming the cursor task and the remaining count. The nudge is UNGATED from `continuous_mode` — it fires whenever there's pending plan work, because the task tool represents an explicit operator-approved plan. It is bounded by `autonomous_loop_limit` (the same cap continuous mode uses) and by the `_task_nudge` inbound marker that prevents two consecutive nudges from the same cycle. The ledger is sticky across `/reset` — `clear_conversation` does not wipe tasks, only `task clear` (or `clear_tasks`) does.
**Consequences:** Operator plans survive across arbitrarily many turns in both continuous and single-turn modes; the model's active work is always visible in the prompt it's reading. The nudge makes "declares victory mid-plan" self-correcting without requiring continuous mode, which matters because continuous mode was conceptually downgraded in v0.7.0. Tradeoffs: (1) the ledger adds one INTEGRATE-time SQLite read per cycle regardless of whether tasks exist (cheap, but non-zero); (2) the nudge fires even when the model was legitimately waiting for user input — mitigated by the `_task_nudge` single-fire guard and the operator's ability to call `task clear`; (3) semantic-grain tasks are an operator-facing contract — if the model creates tasks at tool-call grain instead, the ledger bloats, so the `task` tool description explicitly teaches "one task = one milestone"; (4) this re-introduces a plan-shaped structure that v0.7.0 removed, but deliberately at a lower altitude — no scheduler, no background firing, no role binding, just a user-approved sequence of milestones the model is expected to work through.

### D-20260419-13 — Track Loop Step Attribute on CoreLoop Instance
**Date:** 2026-04-19
**Status:** Accepted
**Context:** D-20260419-10 added the `if self.step != LoopStep.OBSERVE` guard inside `submit_input` so that conversational submits would not inflate `user_interrupts_total`. The guard read `self.step`, but no code path ever set that attribute — `_set_step` only emitted the `step_changed` Qt signal, relying on listeners to track the current step externally. The guard threw `AttributeError` the first time any user typed input. The bug was masked in development because the error surfaces only on the interrupt path (cancel_event already set when submit_input runs), and was caught in production when Kevin submitted input mid-cycle.
**Decision:** Initialize `self.step = LoopStep.OBSERVE` in `CoreLoop.__init__` and have `_set_step` assign `self.step = step` before emitting the signal, so the attribute is always a valid mirror of the current step independent of signal delivery timing. `submit_input`'s interrupt-count guard now reads a value that is guaranteed to exist and reflects the step the cycle is actually in.
**Consequences:** The interrupt telemetry fix from D-20260419-10 now works as intended. No downside — the attribute write is O(1) and cost-free; tests and other callers that want to introspect the current step can read `self.step` without waiting for the Qt event loop to deliver the signal.

### D-20260419-14 — Refine Task-Nudge Single-Fire Suppression to Non-Continuous Mode Only
**Date:** 2026-04-19
**Status:** Accepted (refines D-20260419-12)
**Context:** The v0.8.0 task-ledger nudge (D-20260419-12) used a blanket `_task_nudge` inbound marker to prevent two consecutive nudges from the same cycle, bounded "by autonomous_loop_limit + _task_nudge single-fire guard". ### D-20260417-07 — Log summarizer must segregate incidents from routine
**Date:** 2026-04-17
**Status:** Accepted
**Context:** The first two digests produced by `tools/log_summarizer.py` omitted real ERROR-level events and hallucinated a "clean system state." Investigation traced the miss to `_build_prompt`: (a) entries were rendered in a single chronological stream, so the ~1% of ERROR lines were drowned out by ~99% of routine `loop.contextualize` INFO chatter, and (b) context dicts were reduced to key names only (`[ctx: exception, path]`), which stripped the actual error details — exception class, file path, traceback — before the model ever saw them. On top of the miss, the model editorialized one of the path errors as a sandbox-escape attempt, which was wrong: the agent was holding a stale legacy path from before the v0.4.0 consolidation.
**Decision:** Change the prompt structure so incidents can't get lost in the noise. Split the entry stream into two labeled sections — `INCIDENTS` (WARNING/ERROR/CRITICAL with full context values, clipped only per-value at 400 chars) and `ROUTINE` (INFO/DEBUG with component + message + context key names only). Add hard rules in the prompt: (1) lead with incidents, (2) quote context values verbatim, (3) group repeated errors with a count rather than enumerating, (4) cap routine coverage at two trend-level bullets, (5) forbid speculation about agent intent (a path error means the path was wrong, nothing more).
**Consequences:** Error details (paths, exception classes) now survive into the prompt and therefore into the digest. Routine cycle chatter still gets compressed, so the prompt size is bounded — the routine block is capped at ~20k chars; the incidents block is never truncated because it's the point. Prior digests are left in place per the append-only rule; a Corrective Note was added to `codex/log_digest.md` documenting what was missed.

### D-20260417-08 — Log summarizer must send log data as a user turn, not as a system prompt
**Date:** 2026-04-17
**Status:** Accepted
**Context:** Even after D-20260417-07 segregated incidents from routine, the log digest kept arriving empty when run against gemma:26b. Switching to llama3:9b for the same call succeeded. Investigation: the first revision of `_summarize` packed the entire prompt (rules + log data) into `OllamaClient.chat(system_prompt, messages=[])`. `OllamaClient.chat` then sent Ollama a request whose only message was a `system` turn with no `user` turn. Gemma and many other models treat that as "system established, now waiting for the human" and return an empty string. Smaller / more permissive models like llama3:9b happily answer anyway, which is why that model worked in isolation. A secondary contributor was a 60s `timeout` — a 26B model processing ~12k characters of input regularly takes 60–120s locally, so even when the request was shaped correctly, short timeouts produced empty responses.
**Decision:** Split the prompt into `system_rules` (scaffolding: 5 HARD RULES) and `user_content` (the labelled INCIDENTS + ROUTINE log sections). Call the client as `client.chat(system_rules, [{"role": "user", "content": user_content}], timeout=300)`. Cap the user payload at ~12k characters so it fits in a default 8k-token window. Add an empty-response guard at the end of `_summarize` → `execute`: if the model still returns an empty string, surface it as an explicit error and refuse to advance the checkpoint or write a digest entry.
**Consequences:** The tool now works against gemma:26b as originally intended, independent of which Ollama model is loaded. `num_ctx` does not need to change — the 12k-char cap was chosen to fit the stock 8k-token window. Silent failures are impossible: either the digest contains a real model-written summary, or the caller sees an explicit "empty response" error and the checkpoint stays put so the next run tries again. The design fully decouples the tool from the prompt-conventions of any single model, which matters because `_detect_loaded_model` will pick up whatever is currently hot in Ollama.

### D-20260417-09 — All tool path arguments are project-root-relative (single-anchor resolver)
**Date:** 2026-04-17
**Status:** Accepted
**Context:** Sentinel was repeatedly hitting FileNotFoundError on hallucinated absolute paths — not on legacy pre-v0.4.0 folder names (as initially theorized), but on mangled user segments produced freshly each cycle: `C:/Users/ke/OneDrive/Desktop/ai/core/tool_registry.py`, `C:/Users/iam/OneDrive/...`, `C:/Users/kevin/OneDrive/to/Desktop/ai/...`. Two reinforcing prompt defects drove the behavior. First, `core/loop.py` substituted `{workspace_folder}` and `{codex_folder}` as **absolute** Windows paths into the `[WORKSPACE POLICY]` block, training the model to imitate the absolute form in tool arguments. Second, `tools/analyze_directory.py`'s schema description literally contained the string `'C:/Users/kevin/OneDrive/Desktop/ai'` as an example. Every cycle, the model saw absolute paths in the prompt and emitted (usually-mangled) absolute paths in its tool calls. Antigravity proposed a fuzzy resolver (substring-match on workspace folder name, suffix-match against a cached file list) to silently correct the bad paths. That approach was rejected: (a) "ai" as a substring collides with legitimate filenames, (b) the user segment is unbounded so there's nothing finite to match against, (c) silent correction hides the drift signal from the Sentinel log that exists specifically to surface this class of bug, (d) suffix-match introduces non-determinism (resolution depends on what other files exist at call time). **Decision:** Single-anchor discipline. A new `core/path_utils.py` exposes `resolve(path_arg) -> Path` with a hard contract: reject absolute paths (POSIX or Windows drive-letter) regardless of host OS, reject `..` escapes, join relative paths to `PROJECT_ROOT` (the one absolute anchor, derived at import time from `Path(__file__).parent.parent`). `PathRejectedError` carries a model-readable message that names the correct shape. `tools/filesystem.py`, `tools/analyze_directory.py`, and `tools/screenshot.py` route their path arguments through `resolve()` as the first step of `execute()`, returning `f"Error: {e}"` to the model on rejection so the error lands in the next cycle's context. The system prompt is corrected at the source: `{workspace_folder}` and `{codex_folder}` now render as relative strings (`workspace/<model>`, `codex`), the `analyze_directory` schema description uses relative examples, and a new `[PATH DISCIPLINE]` block spells out the rule with CORRECT/REJECTED examples. **Consequences:** Mangled-user-segment paths can no longer land in the filesystem layer. The prompt and the tool contract are now consistent — the model sees only relative paths in every context block AND is only allowed to emit relative paths, so imitation and permission agree. Sandbox is strictly tighter than before: previously, a correctly-formed absolute path that happened to begin with the project root was accepted; now nothing absolute is accepted at all. Trade-off: we accept that the model must emit paths in the exact shape we demand, with no salvage. That's the whole point — the rejection error is the teaching signal, and the summarizer can count rejections if the model is failing to internalize the rule.

### D-20260418-01 — filesystem gains move, delete, and block pagination
**Date:** 2026-04-18
**Status:** Accepted
**Context:** Three gaps in `tools/filesystem.py` were making normal maintenance awkward. (1) Archival workflows (Scholar's closed-proposal sweep, Architect's old-review rollover) had no in-tool way to relocate files — the model was falling back to `shell_exec mv`, which is sandbox-wider and leaves a less-auditable trace. (2) There was no way to remove a single stale file without a shell call. (3) Files larger than the tool registry's `MAX_TOOL_OUTPUT=16000` char cap got silently clipped at the tail on `filesystem:read`, losing content the model couldn't recover without a secondary tool (and losing any footer hint about what was cut).
**Decision:** Extend the existing tool with three surgical additions. `operation: "move"` takes a `dest` argument, routes both endpoints through the single-anchor resolver, refuses to overwrite an existing destination (archival must never silently destroy data), and returns a relative-form `Moved 'A' -> 'B'` confirmation. `operation: "delete"` refuses directories (no `rm -rf` footgun; empty-then-delete is the audit-trail pattern) and only removes single files. `operation: "read"` gains an optional `block` integer parameter that paginates large files in 15000-char chunks — deliberately below the 16000-char registry cap so the `[BLOCK N OF M — chars X..Y of Z]` footer fits under the clipping threshold without being truncated. Block auto-triggers for files over one block (even on `block=0` default) and explicitly respects any caller-provided block index for later chunks; the footer always names the exact `block=N+1` value to pass for the next call.
**Consequences:** Model has first-class in-tool verbs for archival and for navigating large files, and the operations compose cleanly (move → delete → read with block=N). Overwrite refusal is a hard rule, not an advisory — roles that want to replace a destination must delete explicitly first, which makes the log a reviewable record. The 15000-char block size is the load-bearing number: if `MAX_TOOL_OUTPUT` ever changes, block size should track it with a footer-sized margin. Legacy `max_lines` path for small files is preserved for backward compatibility with callers that paginate by lines.

### D-20260418-03 — One malformed goal must not masquerade as an empty queue
**Date:** 2026-04-18
**Status:** Accepted
**Context:** A hand-edited entry in `goals.json` had `"schedule_minutes": "0"` (string, not integer). `_check_goals_status` computed `sched_s = v.get("schedule_minutes", 60) * 60`, which in Python evaluated `"0" * 60` as string repetition — a 60-character string of zeros. Future goals.json edits that drift into string types get a named error at the exact entry, not a whole-queue stoppage. Trade-off: we accept that the loop silently skips malformed entries rather than refusing to start — this matches the existing pattern for `schedule_minutes <= 0` and keeps the queue available for the entries that are still valid.

**Consequences:** The version sequence is monotonic regardless of archival order. A cycle that reverses step 3 or crashes mid-cycle self-heals — the next scholar_runner run sees the archived `v<N>` in `old_stuff/`, returns `next_version = N+1`, and the Scholar picks up where it left off. The model no longer does filename arithmetic, removing a whole class of "v+1" mis-parses. A second trade-off: in the warning branch, `last_review_update` is `null`, so the delta scan uses mtime 0 as its threshold — every file in the tree becomes a delta. This is intentional: when the baseline is unrecoverable the Scholar needs to treat the whole tree as "since unknown", and the expense is bounded by the single-shot nature of the recovery.

### D-20260418-07 — Architect and Analyst auto_tools carry pre-fetched target context; archival guard is a first-class step
**Date:** 2026-04-18
**Status:** Accepted
**Context:** Four consecutive Architect cycles (CP-20260418-02/-03/-04/-05; see `rejected_proposals.md`) drafted proposals against archived files — a typo fix in `architecture_review_v6.md` while v8 was live, ADR backfills into `architecture_review_v5.md`, and a label change against a Sentinel manifest that already carried the correct label. Each matching Analyst critique APPROVED, because its auto_tool (`filesystem:list` on the workspace folder) carried no signal about whether a named target was archived or superseded. Two teaching defects combined into one failure mode. First, the Architect's auto_tool (`scholar_runner` with default args) returned only `review_path` — a string — so the model either burned a cycle re-reading the file, or worse, worked from its recollection of a prior review. When memory drifted, so did the target pointer. Second, the Analyst had no way to know that `architecture_review_v5.md` was superseded by v8 without computing the baseline itself; in practice it approved the stale-target proposals at face value. The shared structural fix is to make each role's auto_tool carry the context it actually needs, and to write the archival check into both manifests as a named step.
**Decision:** Two complementary changes. (1) `tools/scholar_runner.py` gains an opt-in `include_review_head: bool` parameter that inlines the current architecture review's text into the payload, capped at `_REVIEW_HEAD_MAX_CHARS=8000` with a `[...TRUNCATED — N more chars; issue filesystem:read on <path> block=2 for the tail]` sentinel. `roles.json` sets `include_review_head=true` on the Architect's auto_tool so the baseline review lands in its nudge every cycle; the Scholar's auto_tool leaves the flag off because the Scholar is about to rewrite the review anyway. The Architect manifest is rewritten to teach the model to anchor on `review_head` and explicitly forbid working from memory of a prior version. (2) A new `tools/analyst_runner.py` replaces the Analyst's `filesystem:list` auto_tool. It auto-picks the workspace (or honors an optional `workspace_folder` arg), locates the newest `change_proposal_CP-*.md` without a matching `critique_CP-*.md`, inlines the proposal body up to ~8000 chars, extracts every referenced file path from the proposal text, previews each target (up to 8 targets, 1500 chars each), and classifies every target as `is_archived=true/false` via two rules — any path whose segments contain `old_stuff/` is archived, and any `architecture_review_v<N>.md` with N below the highest active version is archived-superseded. The payload ships `workspace_folder`, `target_proposal`, `proposal_text`, `target_files[{path, exists, is_archived, archived_reason, preview, total_chars}]`, `warnings`, `error`. Both manifests (Architect and Analyst) gain an explicit "ARCHIVAL GUARD" step immediately after reading the auto-tool payload: for the Architect it forbids drafting proposals targeting `old_stuff/` paths or superseded reviews; for the Analyst it mandates a REJECT verdict (with rationale `"targets archived file — <path> is superseded (<archived_reason>)"`) when any target entry has `is_archived=true`.
**Consequences:** The Architect sees the live baseline review text in every nudge, so "wait, which architecture_review am I looking at?" becomes a non-question. The Analyst sees each target file's archive status and preview before issuing a verdict, so a stale-target proposal is rejected at its first critique pass rather than approved. Both roles have a named guard step that Kevin (or any future reviewer) can point to when a critique goes wrong. Trade-offs: (a) `review_head` adds up to 8KB to the Architect's nudge — acceptable, since the nudge is transient and a fresh baseline avoids an extra `filesystem:read` cycle that was costing more tokens than it saved. (b) `analyst_runner` path extraction is a heuristic (regex over file-extension suffix list + reject substrings for URLs and shell paths, plus a pre-context guard that drops any match preceded by `/` or `:` to defeat URL-scheme false-positives where the regex cuts at the colon before the hostname); false negatives are still possible when a proposal references a target by partial path or by prose description. Monitored — the tool surfaces a `warnings` field when extraction looks suspicious so the human reviewer sees the gap before the Analyst commits to a critique. (c) `analyst_runner` caps at 8 targets per proposal; proposals that genuinely touch 8+ files are rare in practice, and the cap is a cheap guard against runaway payload size. If it becomes a constraint, bump `_MAX_TARGETS` and `_TARGET_PREVIEW_CHARS` together so the total payload stays under the registry's 16000-char cap.

### D-20260418-09 — [ACTIVE GOALS] prompt rendering must be gated on continuous_mode so role goals don't auto-elect on user turns
**Date:** 2026-04-18
**Status:** Accepted
**Context:** Kevin reported: "it shouldnt run through roles when continuous mode is not checked, its unchecked by default." The observed symptom was that on reboot, asking Servo a plain user question caused it to invoke `scholar_runner` (the Scholar's auto-tool) and attempt a Scholar cycle, stopping only when `chain_limit` capped the chain. Investigation in `core/loop.py` confirmed that the continuous-mode gating at the execution layer is structurally correct: `continuous_mode` defaults to False (line 55), the GUI checkbox defaults unchecked, `_active_role` is cleared on every user submit (line 115), the goal-election block is gated by `elif self.continuous_mode:` (line 339), `_active_role` is only ever set inside that gated block (line 395), and `_check_goals_status` on the idle path only expires finite goals and emits `goals_changed` — it does not fire nudges. The leak was one level above the execution layer: in the prompt itself. `_build_system_prompt` unconditionally renders every continuous role goal whose `time_since >= sched_sec` as `[CONTINUOUS PRIORITY 2] role_scholar: [Scholar] ... (DUE NOW - Please execute and call goal_manager mark_done)`, with a `CRITICAL: You must execute ...` header above the list. When the user sent any message on reboot, the system prompt handed the model an explicit instruction to run the due role, and the model did exactly that — not because continuous_mode was True (it wasn't) but because the prompt said the goal was due and told the model to execute it. The chain_limit cap (loop.py:163) is what stopped the cascade; absent that, the same language would have prompted a full role cycle on every user turn.
**Decision:** Make the `[ACTIVE GOALS]` block in `_build_system_prompt` mode-aware. Split goals into `finite_items` and `continuous_items`. Three branches on `self.continuous_mode`:
(1) `continuous_mode == True` — legacy behavior preserved: CRITICAL-finite-before-continuous header; continuous goals render as `DUE NOW - Please execute and call goal_manager mark_done` when due, `Snoozing for N minutes` otherwise.
(2) `continuous_mode == False` AND finite goals exist — CRITICAL header narrowed to finite goals only, with explicit lockout language: `Continuous Mode is OFF — continuous role goals below are listed for visibility only; you MUST NOT elect yourself into one on a user turn or invoke their auto-tools unprompted.` Continuous goals render with an inert label: `Scheduled every N min — Continuous Mode OFF, will not fire`. No `DUE NOW`, no `Please execute`.
(3) `continuous_mode == False` AND only continuous goals exist — CRITICAL header dropped entirely; lockout language stands alone. Continuous goals still render inertly for visibility.
The `_goal_description` helper is unchanged; the live-role-task lookup behavior from D-20260412-01 (title + task from roles.json) is preserved in every branch. The `schedule_minutes` parse inside the rendering loop also gets the same int-coercion guard that `_check_goals_status` uses (D-20260418-04), defaulting to 60 on a malformed entry so one bad goal does not kill the whole block.
**Consequences:** When continuous_mode is off (the default), the model sees a goals block that describes scheduled work without ever instructing action. The Scholar-on-reboot failure mode Kevin observed cannot happen from this vector — the prompt itself refuses to ask for it. Visibility is preserved so Kevin (and the model, for transparency) can still see what is scheduled and why it is quiet. The legacy wording is restored verbatim when continuous_mode is on, so a user toggling the checkbox mid-session sees identical behavior to the v0.6.5 regime. Trade-offs: (a) Three possible headers instead of one — a small increase in prompt branching that is justified because the lockout-off-continuous-only case needs different wording than the lockout-with-finite-tasks case (a model executing a finite task wants to know its priority relative to the silent continuous goals below it). (b) The inert label `(Scheduled every N min — Continuous Mode OFF, will not fire)` is slightly longer than the old `(DUE NOW ...)` / `(Snoozing ...)` forms, which adds a handful of tokens per continuous goal. Acceptable — continuous goals are five entries in Kevin's current setup, so the overhead is ~30-50 tokens and buys a robust lockout. (c) This patch does NOT touch `[ROLE BRIEFING]` or the active-overlay block; those are already gated on `_active_role != "servo"`, and `_active_role` is correctly cleared on `submit_input`. If a future regression reintroduces a role-briefing-on-user-turn leak, investigate the `_active_role` lifecycle, not this block. Synthetic coverage: `test_goals_block.py` in outputs/ asserts 17 properties across four modes (continuous on + continuous-only; continuous off + continuous-only; continuous off + mixed; empty queue in both modes). All 17 assertions pass.

### D-20260419-01 — INTEGRATE-time auto-compression of conversation_history via [PRIOR CONTEXT] block
**Date:** 2026-04-19
**Status:** Accepted
**Context:** With the Phase 1 kernel shipped (D-20260418-10), the next consumer in Kevin's assessment brief was "directed tool that fires as part of the loop ... in integrate to clean up after a Reason Act". Conversation_history is a turn-count cap (default 15, bounds 3-30, tunable via system_config; reduced by 2 per Critical hardware cycle per D-20260418-04's throttle logic). Today that cap is the ONLY control on how much transcript the model sees: once the cap is hit, older turns silently fall off the end, which means the model forgets user requests, tool outcomes, and commitments it made. Raising the cap puts pressure on the 8k-token context window (roughly ~32k chars total, leaving headroom for system_rules + response). A conversation over, say, 50 turns in a single session today is either (a) only remembered at depth 15 or (b) if the cap was raised to 30+, chewing half the context window in every REASON call. Four design alternatives were considered: (1) collapse oldest raw turns into a synthetic role='summary' row that sits inside the user/assistant message stream, (2) keep raw turns untouched in SQLite and inject a summary as a [PRIOR CONTEXT] block alongside [IDENTITY] / [ROLE BRIEFING], (3) compress based on char budget instead of turn count, (4) ship behind a GUI toggle that defaults off. Kevin picked (2) + (the 2×cap trigger option) + (no toggle) — all three recommended defaults. Option (2) was strictly better than (1) because it keeps the full audit trail recoverable and it fits the existing prompt-block pattern rather than introducing a novel "synthetic turn" concept the model might try to treat as a real turn. Char-budget triggering would have tied compression cadence to raw content length rather than to the user's tuning of conversation_history, which would drift from the hardware_policy throttle; turn-count triggering scales naturally with the cap.
**Decision:** Ship Phase 2 as a non-destructive, INTEGRATE-time compressor: raw turns stay in the `conversation` table untouched; a new `conversation_summary` table tracks the live "prior context" summary and the range of conversation ids it covers. On every non-transient INTEGRATE, `core/history_compressor.maybe_compress(state, conversation_history)` is called. Trigger: uncompressed raw turns (conversation rows with `id > last_summary.covers_to_id`) ≥ 2 × `self.conversation_history`. When the trigger fires, the compressor rolls up everything older than the newest `conversation_history` turns into a new summary row, absorbing the prior summary's text so only one "live" summary ever exists (older rows kept for audit). The kernel is the shared `tools.summarizer.summarize` from Phase 1 — this consumer supplies its own `system_rules` block (conversation-specific: preserve Kevin's recent requests verbatim, preserve tool outcomes + decisions + commitments, strip reasoning chains and verbose tool payloads, write one narrative paragraph rather than bullets) and handles the `(summary, model_used)` return. Wiring touches four points in `core/loop.py`: (a) CONTEXTUALIZE loads the latest summary into `context["history_summary"]` next to history; (b) `_build_system_prompt` renders a `[PRIOR CONTEXT]` block between `[ROLE BRIEFING]` and `[SYSTEM ENVIRONMENT]` with explicit "treat this as your own memory" framing so the model doesn't ask Kevin to repeat decisions captured in it; (c) `_build_messages` filters out history turns whose id ≤ `summary.covers_to_id` so the model doesn't see the raw turns AND the summary in the same call (which would double-count the signal and waste the context win); (d) INTEGRATE calls `maybe_compress` after the persist block, wrapped in an exception guard that logs-and-continues so the compressor can never crash the loop. Failure semantics match the Phase 1 kernel contract: empty response or kernel exception → log WARNING/ERROR, no summary saved, `compression_last_failed_turn_count` state key set to the current uncompressed count so the trigger waits `+cap` more turns before retrying (prevents retry-storm on a pathological payload). Coverage: `tests/test_history_compressor.py` — 17 tests across four classes (`TestShouldCompress`: 6 predicate boundary cases, `TestConversationSummaryTable`: 6 round-trip and CRUD, `TestMaybeCompress`: 5 end-to-end with kernel mocked, `TestBuildMessagesFilter`: 3 filter cases including legacy-no-id rows). Predicate sanity passed in isolation (11 assertions covering 2× cap, backoff math, throttle-tight-cap edge); full suite validates in Kevin's environment.
**Consequences:** The model gains a continuous-memory prompt surface that grows proportionally with usage instead of silently dropping old turns at the cap. A 50-turn session with `conversation_history=15` now sends the last 15 raw turns plus a narrative paragraph covering turns 1–35, rather than just turns 36–50. The paragraph lands in the system prompt where the existing block machinery ([IDENTITY], [ROLE BRIEFING]) already shapes the model's self-understanding, so there's no novel "where does this text go" decision for the model to work out. Trade-offs: (a) Each INTEGRATE runs a cheap COUNT + SELECT MAX even when compression is not due — negligible SQLite cost, but worth noting; the actual kernel call (the expensive part) only fires when the trigger is hit, which at default cap is once per ~30 turns. (b) The compressor's prompt content is a hand-written block in `core/history_compressor.py`; the Phase 1 ADR established that each caller owns its rules. A future change to "always quote the last user request verbatim" lands in one file. (c) covers_from_id is recorded for audit/diagnostics only; the runtime filter uses covers_to_id exclusively. If a debugger wants to know "when did the summary containing <phrase> first appear?", prior rows in conversation_summary are kept rather than updated in place. (d) The `[PRIOR CONTEXT]` block framing ("treat it as your own memory of those turns — do not ask Kevin to repeat decisions or requests captured here") is load-bearing. Without it, early testing on gemma 26B showed the model would occasionally treat the block as a user-supplied recap and ask clarifying questions about it. The current wording reads as self-referential memory and stops that failure mode. (e) Hardware_policy's throttle (D-20260418-04 et al.) and Phase 2 compression both write `self.conversation_history`; they are independent — throttle is reactive to hardware pressure, compression is proactive to history age. They coordinate by chance (throttle fires in REASON, compression in INTEGRATE; second writer wins) but there is no race because the loop is single-threaded. A throttled cap of 7 means compression fires at 14 uncompressed turns instead of 30 — the behavior scales correctly under pressure. (f) Phase 2 is belt-and-suspenders, not load-bearing: if the kernel times out, the empty-response branch keeps the raw turns and just logs the failure. No user-visible degradation. (g) Backoff after failure waits +cap turns (so at default cap, retries wait ~15 more turns). That is long enough to avoid retry-storm on a pathological 12k-char payload that consistently times out, but short enough that transient issues (Ollama paused, network blip) recover quickly. (h) This ADR closes the Phase 2 scope; Phase 3 (CONTEXTUALIZE-time file-payload compression) and Phase 4 (agent-facing `summarize` tool) are future work with separate ADRs.

The duplication is at the display layer. `gui/log_panel.py::LogPanel` has two independent ingress paths: (a) `on_log_event` slot, wired to `CoreLoop.log_event` for low-latency rendering, (b) `QFileSystemWatcher` on `sentinel.jsonl` that detects file growth and parses appended lines in `_on_file_changed`. Both end at `_render_entry`. Every `_slog` call hits the widget twice — once via signal, once via watcher. Every `state.add_trace` mirror (INFO `loop.<step>` lines) also hits the watcher (it has no signal path), but because the watcher path is the ONLY path for those entries, they would render once… except the watcher was firing duplicately on at least some writes, which combined with the signal duplication made WARNING and INFO both double visibly. Dropping the file watcher entirely would fix the WARNING dupe but would silence all entries that bypass `_slog` (`state.add_trace`, `core/tool_registry.py`, `core/history_compressor.py` — anywhere a module calls `get_logger().log(...)` directly, of which there are several). Dropping the signal would lose sub-second latency and the context_json payload. Neither is acceptable on its own.
**Decision:** Keep both ingress paths for resilience (signal for latency, watcher for coverage of non-signalled entries) and dedupe at render time by `(level, component, message)` within a 2-second recency window. Implementation: `LogPanel.__init__` gains `self._recent_renders: deque(maxlen=64)` of `(key, monotonic_time)` pairs. `_is_duplicate(level, component, message)` prunes entries older than the 2s window, checks for a content match among survivors, records a miss, and returns hit/miss. `_render_entry` calls `_is_duplicate` immediately after the level-filter check and before HTML rendering, returning early on match. `_refresh_display` (which fires when the user toggles a level filter and re-renders the full buffer) explicitly `_recent_renders.clear()`s so a legitimate historical stream of identical entries doesn't self-suppress during the refresh. The 2s window was chosen because: (a) signal and file-watcher races for the same event always land within tens of milliseconds of each other on a healthy filesystem; (b) a real duplicate of two actually-distinct events with byte-identical `(level, component, message)` and arriving less than 2s apart is vanishingly rare in this codebase (the only candidate is a rapid-fire tool failure, which would still be visible via the second one appearing after the window); (c) widening to 5s or more would start suppressing legitimate back-to-back events on continuous-mode scholar cycles.
**Consequences:** The display lane now matches the on-disk lane one-for-one for every event — the widget renders a single line per sentinel.jsonl line. Diagnostics get clean again: when an operator sees WARNING auto-continuing (1/2) followed by (2/2), they know the truncation fired twice, not once with a display echo. Trade-offs: (a) A true burst of identical events within 2s will show exactly one entry; this is an acceptable loss because (i) such a burst almost always means a retry storm worth investigating via the sentinel.jsonl file (where all entries land) rather than the live console, and (ii) the analytics sparkline uses the singleton logger's `get_error_counts()` which reads the file directly, so suppressed events are still counted in the sparkline. (b) The 64-entry deque + 2s window imposes O(n) dedupe cost per render where n ≤ 64 — negligible relative to the Qt paint cost. (c) Dedupe state is process-local; a log panel opened mid-session starts with an empty tracker and de-dupes from that point forward. That's fine because historical entries are loaded via `_initial_load` which sequentially feeds them through `_render_entry` with an empty tracker, so the tracker fills with the last entries loaded and prevents only the earliest moments of overlap with live events — an acceptable edge case. (d) This does not fix the underlying Scholar slowness (D-20260419-03); that is a separate defect — the 48k-token output across 3 truncated streams is legitimately generated content, not a ghost.

### D-20260419-03 — Scholar writes the architecture review via filesystem:write content, not chat prose
**Date:** 2026-04-19
**Status:** Accepted
**Context:** Kevin observed a Scholar cycle chew 12+ minutes of wall clock in continuous mode, with three Ollama streams each hitting num_predict (16384 tokens) and each running ~260s at gemma4:26b's ~60 tok/s throughput — ~48k tokens of output across the cycle, auto-continue exhausted at 2/2, no visible progress. Reconstructing the sequence from the verbose stream log and `_auto_continue`'s behavior: stream 1 (14.6s preamble) ended with a `filesystem:read` on the first delta file (likely `core/loop.py`). Stream 2 was the response to that tool result — and it burned the full 16384-token budget without emitting any tool call. Auto-continue 1 fed it "continue where you left off" and it burned another full budget. Auto-continue 2 did the same, then the 2/2 give-up fired. The only plausible explanation consistent with the `num_predict`-equals-budget pattern and the lack of a tool call is that the model was **composing the architecture review body directly in its chat response** instead of emitting a `filesystem:write` call with the review as the `content` arg. The `scholar.md` step 3 wording at the time ("Emit the new review. Write workspace/<model>/architecture_review_v<N>.md ... Prefer a coherent narrative over a changelog") was ambiguous between (a) "emit a `filesystem:write` call whose content is the narrative" and (b) "write the narrative in your response." The weaker model read (b). Auto-continue then reinforces the failure: after the first num_predict cap, the continuation prompt is "continue exactly where you left off" — which tells a model that is mid-prose to keep writing prose, not to stop and switch modalities to a tool call. The three-stream cascade follows deterministically. Secondary observation: this is the generalized form of D-20260418-06 (the WRAP-UP nudge was interpreted as a text template rather than a tool-call contract). Both failures come from the same underlying antipattern — a manifest that describes a long-document emission without explicitly locking the tool-call frame.
**Decision:** Tighten `codex/role_manifests/scholar.md` step 3 with an explicit **WRITE-CALL DISCIPLINE** sub-block that makes three rules unambiguous: (1) the review body MUST appear only inside the `content` argument of a single `filesystem:write` call; (2) the chat response for this step is ONE short sentence naming the version about to be emitted (e.g. "Emitting `architecture_review_v9.md`."), not narrative prose; (3) if the model catches itself writing a paragraph of review prose in chat, STOP and fold it into a `filesystem:write` call instead. The sub-block is positioned inside step 3 rather than at the top of the manifest so it lands in the model's attention at the exact moment it is deciding how to emit the review. The guidance is phrased as imperative "MUST / do not / STOP" language because softer phrasing ("prefer a coherent narrative") is what got us here. Scope is deliberately limited to the Scholar — the Architect has a similar shape (writes change proposals) and may need the same treatment, but Kevin's report was Scholar-specific and over-eager manifest rewrites risk breaking stable roles. A followup ADR will generalize if Architect shows the same behavior. Num_predict tuning is Kevin's lever; this ADR does not touch runtime_config.
**Consequences:** The Scholar cycle on the next continuous-mode run should converge to the expected shape: short preamble, per-delta `filesystem:read` calls (one tool call per response), then a single `filesystem:write` emission with the review as `content`, then `filesystem:move` for archival, then `mark_done`. Wall clock drops from ~12 minutes to roughly (reads × 15-20s) + (one write stream) + (one move stream) + (mark_done) — dominated by the write stream, whose token cost is the review body itself rather than the review body × 3. Trade-offs: (a) The manifest now has a chunky imperative sub-block inside step 3 which makes the step visibly longer in the role briefing — a small prompt-token cost (~150 tokens per cycle) that is justified because the failure mode it prevents is a 12-minute wall-clock burn. (b) "Do not narrate the review contents in chat" can be mis-read as "do not explain your work" — mitigated by the explicit allow-list for a one-sentence version-naming preamble. (c) The rule is manifest-only; if a future model ignores role briefings, we would need to fall back to structural enforcement (num_predict cap, content-length heuristic, tool-call-required predicate in the continuation prompt). This ADR explicitly defers those structural options because the manifest fix is cheaper and reversible. (d) Auto-continue's "continue where you left off" prompt remains a latent amplifier of this failure mode across all roles; a future ADR may change the continuation instruction to something like "If your previous response was mid-tool-call, complete the JSON block. If it was prose, emit a tool call now." The JSON-detect branch in `_auto_continue` (core/loop.py:1037-1047) already handles the first half of that; the prose-detect branch does not. Left for a separate fix. (e) This ADR + D-20260419-02 shipped together but address distinct defects: D-20260419-02 is a display-layer bug in `gui/log_panel.py` that made Kevin suspect a doubled truncation; once the display bug is fixed, future Scholar-cycle diagnostics will show the single real truncation events and the manifest fix's effectiveness will be directly observable in the cycle's wall-clock time.

### D-20260419-04 — Phase 3 summarization hooks: filesystem:read `summarize` flag + default-on delta pre-summarization in scholar_runner
**Date:** 2026-04-19
**Status:** Accepted
**Context:** D-20260419-01 shipped Phase 2 (non-destructive INTEGRATE-time compression of conversation_history via the `[PRIOR CONTEXT]` block) and explicitly deferred Phase 3 (CONTEXTUALIZE-time compression of file payloads) and Phase 4 (an agent-facing `summarize` tool). Kevin returned to the deferred work with two shape alternatives for the summarization hook: (A) add a tool-argument lever so the model can opt individual reads into summarization inline — pseudocode `file read block 1 prev=summarize,clean,keep` / `read(filename,1).summarize` — or (B) refine role tasks to issue explicit `summarize(read(file))` chains. Scope review produced three design answers: (1) attach the lever to `filesystem:read` only (not a universal wrapper) because filesystem is the dominant ingest path and the other path-reading tools already emit bounded output; (2) empty-kernel responses fall open to the raw content with an inline marker so callers never get silently empty output (same failure-open contract as D-20260418-10's log_summarizer pilot); (3) the lever defaults to ON for `scholar_runner`'s pre-summarization pass so the feature fires in testing without requiring a role-manifest edit, while remaining off by default on `filesystem:read` so existing callers are unaffected. A fourth question — drop/delete semantics ("acknowledge the read without returning content") — was deliberately excluded from v1: the cleanest implementation is a separate hook (e.g. `acknowledge_only: true`) rather than a `content_mode` enum that cross-concerns summarize and drop, and Kevin's priority was shipping summarize cleanly first.
**Decision:** Ship two reinforcing mechanisms landing together. (1) **`filesystem:read` gains a `summarize: bool = False` argument.** When true, the body (after pagination / max_lines, which run first) is fed through `tools.summarizer.summarize` with a file-type-agnostic `system_rules` block, and the result is wrapped as `[SUMMARY of <path> — <N> lines]\n<summary>\n[END SUMMARY]`. Empty kernel returns fall back to the raw body prefixed with `[SUMMARIZER RETURNED EMPTY — returning raw content]`. Kernel exceptions fall back with `[SUMMARIZER FAILED — <err> — returning raw content]`. Pagination footers (`[BLOCK N OF M …]`, `[Showing first N of M total lines]`) are appended AFTER the envelope so the navigation hint survives. TOOL_SCHEMA documents the flag. (2) **`scholar_runner` gains default-on pre-summarization for delta files >500 lines.** Oversized deltas land in the payload as `{path, summary, raw_line_count, raw_bytes}` dicts — the `summary` string already wrapped in the same `[SUMMARY of … — N lines] … [END SUMMARY]` envelope. Small files (≤ 500 lines) stay as plain path strings. A `summarization_stats` diagnostic block (`files_summarized`, `files_skipped_small`, `files_summarize_failed`, `time_seconds`, `enabled`) reports the pass's behavior. A per-call `summarize_deltas: bool = True` arg lets callers opt out (used by workflows that need exact-string matching). **Manifest teaching:** `codex/role_manifests/scholar.md` step 2 documents the mixed delta shape and teaches that verbatim re-reads go via `filesystem:read` with `summarize: false`; step 4 (closed-proposal sweep) explicitly requires verbatim reads of `decisions.md` / `history.md` / `rejected_proposals.md` because the sweep matches on exact `CP-YYYYMMDD-NN` and `D-YYYYMMDD-NN` IDs that a summary could paraphrase. The auto-tool payload description in scholar.md documents the new `summarization_stats` block. Drop/delete semantics are noted as deferred to a future separate hook if ever needed.
**Consequences:** Scholar's context window no longer bloats on the handful of oversized ledgers that recur in every delta scan — history.md, decisions.md, rejected_proposals.md, and any source file past the threshold land as ~2-4 paragraph summaries wrapped in an explicit envelope instead of 1000+ lines of raw content. Wall-clock cost shifts: the scan now spends kernel time (logged via `time_seconds`) up front so the model's own context is smaller; net effect should be positive because the Scholar's subsequent read passes are replaced by embedded summaries, saving many tool-call round-trips. The read-time flag gives the model a lever inside any role where it knows it only needs a conceptual view. Trade-offs: (a) Summaries can paraphrase IDs and break exact-string matching; mitigated by the sweep-step teaching + per-call opt-out on `scholar_runner` and `filesystem:read`. (b) Pre-summarization burns kernel cycles during scan — failure is swallowed (fall through to plain path) and surfaced in `summarization_stats`, so a broken kernel degrades gracefully rather than blocking the scan. (c) The kernel is now called from three places (log_summarizer, history_compressor, filesystem + scholar_runner) — this is the architectural drift the kernel factoring (D-20260418-10) was designed to absorb; the footprint grew as intended. (d) The 500-line threshold is a bet: too low and small source files get needlessly summarized (loss of fidelity for no gain); too high and the growing ledgers stay huge. 500 was picked so that post-release ledgers land as summaries within one to two releases of accumulated entries while small utility modules pass through verbatim. The threshold is a module-level constant (`_DELTA_SUMMARIZE_LINE_THRESHOLD`) and can be tuned without re-architecture. (e) Phase 4 (agent-facing `summarize` tool) remains deferred because the `filesystem:read summarize` flag and the default-on pre-summarization together cover the foreseeable callers; a standalone tool would duplicate surface area. If a caller emerges that needs to summarize non-file content (e.g. aggregated tool output), Phase 4 can land then. (f) Drop/delete semantics stay out of scope; a later hook can add `acknowledge_only` without reshaping the existing flag. (g) Testing: `tests/test_filesystem_summarize.py` and `tests/test_scholar_runner_summarize.py` stub the kernel via `unittest.mock.patch` so no Ollama process is required; the pagination / max_lines interaction is covered so the footer discipline can't silently regress.

### D-20260421-01 — Runtime Gating and Tuning for Summarization Kernels
**Date:** 2026-04-21
**Status:** Accepted
**Context:** The Phase 2 and Phase 3 summarization features (history compression, tool-result compression, file-read summarization) were designed with hardcoded thresholds (e.g., 2× history cap, 4000-char tool threshold) and target lengths (e.g., 800c, 500c). As the system moved toward diverse models (Gemma 4:26B, Llama 3) and variable hardware, these hardcoded values created a "one size fits none" constraint. Operators needed a way to (a) disable background kernels for pure high-fidelity debugging and (b) tune the aggressiveness of the condensation to match their model's specific attention span.

- `summarize_contextualize` (bool): Master gate for [PRIOR CONTEXT] injection during retrieval.
- `summarize_history_integrate` (bool): Master gate for the background roll-up of old conversation turns.
- `summarize_tool_results` (bool): Master gate for per-turn compression of oversized tool outputs.
- `history_compression_trigger` (int): Floating multiplier for the history cap trigger (replaces hardcoded 2×).
- `tool_result_compression_threshold` (int): Character count trigger for tool results (replaces hardcoded 4000).
- `history_compression_target_chars` / `tool_result_compression_target_chars` (int): Narrative paragraph target lengths.

The `CoreLoop` was modified to fetch these values live from `self.state` at the start of each relevant step (`_contextualize`, `_integrate`), making them hot-swappable via `system_config set` without a loop restart. The `history_compressor` and `tool_result_compressor` modules were refactored to allow these parameters to be passed through from the call site.
**Consequences:** Complete operator control over the system's "memory condensation" behavior. Tuning can now happen in real-time to find the sweet spot between context-window economy and information density. Trade-off: increased complexity in the config layer and an increased risk of a user setting "0" or "99999" and either causing a compression loop or effectively breaking the memory system. Mitigated by robust defaults and the existing `_should_compress` predicate.

### D-20260421-14 — Decomplexification: God-Tools Decomposed into Atomic Primitives
**Date:** 2026-04-21

**Date:** 2026-04-23
**Status:** Accepted
**Context:** Phase C closed the circuit with real Cognate bodies but left four known gaps: observation was a bare file-count (no symbolic resolution), procedural_wins used dummy `[0,0,0,0]` embeddings (exact-match only), the dispatch surface was capped at the Phase C six (`file_read`/`file_write`/`file_list`/`file_manage`/`map_project`/`summarizer`) because the five TOOL_IS_SYSTEM tools wrote to the legacy `state/state.db`, and ToolOutcome serialized full payloads into ChromaDB metadata. - **Step 1 — OllamaClient.embed()**: Added a nomic-embed-text wrapper (`core/ollama_client.py`) with configurable `embed_model` and graceful-degradation (`None` on Ollama-down, dimension-checked at the call site). `_EMBED_DIM = 768` pinned as a module-level constant in `lx_state.py` — migrate-on-change, never mix dims in one collection.

- **Step 3 — ToolOutcome fingerprinting**: `core/lx_outcomes.py` dataclass gained `return_value_sha1`, `stderr_sha1`, `return_value_bytes`, `stderr_bytes`; `.compact_metadata()` returns the fingerprint-only view for procedural_wins rows.
